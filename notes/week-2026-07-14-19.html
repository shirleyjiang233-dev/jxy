01｜Autoware 环境与问题定位

目标环境：Ubuntu 22.04、ROS 2 Humble，通过 VS Code Remote-SSH 在远端主机执行安装与调试。

先判断“命令到底在哪台机器上执行”

本地 Windows

负责 VS Code 界面、浏览器、部分代理软件；不能把它的路径和端口直接当成远端 Ubuntu 的环境。

远端 Ubuntu

终端提示符、APT、ROS 2、Autoware 与 Git 实际运行的位置；代理变量也必须在这台机器上有效。

hostname
echo "$SSH_CONNECTION"
pwd
printenv | grep -i proxy

四层排障思路

层级典型现象检查重点
软件源APT 403、InRelease 或签名错误检查是否仍使用失效的 HTTP 源或旧缓存配置；这类问题不等同于 Autoware 自身故障。
权限Permission denied、Docker socket 无权限确认当前用户、文件权限、用户组与服务状态。
代理Connection refused检查 Git 和环境变量是否残留旧代理，例如旧主机地址与端口。
网络链路TLS timeout、连接被重置用最小请求分别验证 DNS、TCP 443、HTTPS 和 GitHub API。
 
git config --global --get http.proxy
git config --global --get https.proxy
curl -I https://github.com
git ls-remote https://github.com/acados/acados.git
关键认识：代理必须对“正在执行命令的那台机器”可达。Windows 上的代理在另一台电脑或另一系统中运行时，远端 Ubuntu 不能直接使用它的 127.0.0.1。

Markdown 与 VS Code Remote

  • VS Code 预览 Markdown：Ctrl + Shift + V。
  • 侧边预览：先按 Ctrl + K，再按 V。
  • 终端查看：less README.md；退出按 q。
  • Remote-SSH 状态可以通过左下角远程标识、hostname 和 SSH_CONNECTION 共同确认。

02｜RoboBus 系统与日常运维

核心是先识别目标车辆与设备角色，再启动软件，避免连错 Wi-Fi 或控制到其他车辆。

核心设备拓扑

设备IP主要作用
IPC<IPC_IP>车载工控机，运行相关服务并参与数据处理。
ORIN<ORIN_IP>NVIDIA 计算平台，承担自动驾驶核心计算任务。
第二台 IPC<SECOND_IPC_IP>承担独立服务或数据链路任务。

自驾维护终端启动流程

确认车辆核对 Wi-Fi 与目标车辆
智能重连IPC、ORIN、ROS 全部在线
检查版本选择团队指定的稳定版本
启动与观察选服务，关闭 HMI，打开 RViz
操作注意：GNSS 指标低于团队有效性阈值时不作为可靠参考；任何启动操作前都先确认车辆与网络。

网络状态检查

ping -c 4 <ORIN_IP>
hostname
ip -br addr
ip route

ping 检查连通性；ip -br addr 快速查看网卡地址；ip route 判断数据从哪张网卡和哪个网关发送。

NoMachine、SSH 与 SCP

  1. 用 NoMachine 远程进入 IPC，并在车辆运行时录制屏幕。
  2. 在本机通过 SSH 登录 IPC，使用 pwd、ls -lh 确认文件路径和大小。
  3. 退出远端终端，在本机终端执行 SCP，把视频拉到本地。
ssh ipc@<IPC_IP>
pwd
ls -lh ~/Videos
exit

# 在本机执行；带空格的文件名要加引号
scp "ipc@<IPC_IP>:~/Videos/<录屏文件>.mp4" .

安全删除原则

ls -lh                     # 删除前先核对
rm -i 文件名               # 交互确认后删除文件
rmdir 空文件夹             # 只删除空目录
rm -ri 文件夹              # 递归且逐项确认，谨慎使用
cd ..                      # 返回上一级
GNSS 与 PPS：GNSS 提供定位和高精度时间来源；PPS 每秒给出一次精确脉冲，用于让相机、雷达、工控机等设备尽量使用同一时间基准。

03｜Failcase、Lichtblick 与软件安装

排查自动驾驶异常时，不只看最后车辆表现，而是沿数据链追踪“最早从哪里开始出错”。

Failcase 判断逻辑

感知问题

障碍物漏检、误检、类别或位置错误。重点关注影响行驶的近处目标，例如三轮车或公交车局部特征。

规划问题

感知结果正确，但路径、速度或避障决策不合理。

控制问题

规划指令合理，但车辆转向、加速或制动执行偏差。

定位问题

车辆位姿、地图匹配或坐标关系异常，进而影响后续模块。

判断原则：找到时间线上“最早出现错误的模块”，而不是只根据最后的急刹、偏航等结果下结论。

查找与检查录制数据

find ~/ -type f \( -name "*.mcap" -o -name "*.db3" \) 2>/dev/null
ros2 bag info <数据包路径>

Lichtblick 常用面板：

  • Topics：查看数据包内有哪些 ROS 话题。
  • Problems：查看解析和数据异常提示。
  • Raw Messages：检查消息字段的原始值。
  • Image / 3D：查看相机画面、点云和空间目标。
  • Plot：观察速度、转角、加速度等变量随时间变化。

Linux 下载与安装软件

wget <下载地址>
curl -LO <下载地址>
sudo apt install ./软件包.deb
sudo dpkg -i 软件包.deb
sudo apt --fix-broken install

优先使用 apt install ./package.deb，因为它能同时处理依赖；dpkg -i 遇到缺少依赖后，再用 apt --fix-broken install 修复。

物联网卡续费检查

  1. 确认车辆、卡号、套餐、到期时间和下月是否继续使用。
  2. 整理待续费清单，避免漏卡或重复续费。
  3. 按公司流程提交申请并拟定续费合同。
  4. 完成后登记续费结果和新的有效期。

04｜Docker、DDS、CycloneDDS 与 Tailscale

这一部分连接了容器运行环境、ROS 2 通信中间件与远程组网。

Docker 权限与进入容器

groups
ls -l /var/run/docker.sock
docker version
cd ~/docker
ls -l scripts
./scripts/docker_into.sh
  • 出现 Docker socket 的 permission denied 时，先检查当前用户是否属于 docker 组。
  • 脚本说明与实际文件名可能不同；先用 ls 核对。实际进入脚本是 docker_into.sh。
  • 进入容器后仍需确认 ROS 2 和 CycloneDDS 环境变量，而不是只看“进入成功”。
echo "$RMW_IMPLEMENTATION"
echo "$CYCLONEDDS_URI"
printenv | grep -E 'RMW|CYCLONE'

DDS 与 CycloneDDS 是什么

概念含义
DDSROS 2 底层使用的数据分发标准，负责节点发现、发布/订阅和通信质量策略（QoS）。
CycloneDDSDDS 标准的一种具体实现；ROS 2 通过 RMW 层调用它完成网络通信。
RMWROS Middleware 接口层，让 ROS 2 可以切换不同 DDS 实现。
QoS控制可靠性、历史深度、耐久性和实时性等通信行为。

“绑定网卡”是告诉 CycloneDDS 使用哪张网卡发送和接收 ROS 2 数据，避免它错误选择 Wi-Fi、Docker、Tailscale 或其他虚拟网卡。绑定错误时，节点可能互相发现不到，或出现数据不通、丢包和延迟。

理解链路：ROS 2 节点 → RMW 接口 → CycloneDDS → 指定网卡 → 其他设备上的 ROS 2 节点。

Tailscale 排查

tailscale status
systemctl status tailscaled
ls -l /var/run/tailscale/tailscaled.sock
sudo tailscale up --authkey <你的密钥>
  • 客户端命令依赖 tailscaled 后台服务和 socket；socket 不存在时，单独执行 tailscale up 不会成功。
  • 避免 Snap 与 APT 两套 Tailscale 同时安装，以免服务、路径和 socket 冲突。
  • Auth Key 属于敏感凭据，不要写进知识库、脚本、截图或公开仓库；如果泄露，应立即撤销并重新生成。

05｜TF、TF 树与 MCAP

TF 解决“各传感器和车辆坐标之间是什么关系”，MCAP 解决“如何把运行中的多类 ROS 数据完整保存下来”。

TF 与 TF 树

TF 用来描述两个坐标系之间的位置和平移关系以及姿态和旋转关系。多个父子坐标系连接后形成 TF 树。

坐标系作用
map全局固定参考坐标系，通常位于 TF 树较高层，用于表达车辆在地图中的位置。
odom局部连续坐标系，短时间运动连续，但可能逐渐漂移。
base_link车辆主体参考坐标系；具体原点由车型定义，常位于车体中心或后轴中心附近。
传感器坐标系相机、激光雷达、IMU 等相对 base_link 的安装位置和方向。

TF 树不一定永远由同一个坐标系作为最大父节点，但自动驾驶常见层级是 map → odom → base_link → sensors。

三轴加速度与三轴角速度

  • 三轴加速度：沿 X、Y、Z 三个方向的线性加速度，反映前后、左右、上下运动变化。
  • 三轴角速度：绕 X、Y、Z 三个轴的旋转速度，常对应 roll、pitch、yaw 方向变化。

MCAP

MCAP 是适合机器人和自动驾驶数据的日志容器格式。它可以在同一份录制中保存：

传感器

相机图像、激光雷达点云、IMU、GNSS。

车辆状态

速度、转角、加速度、底盘反馈。

算法结果

定位、TF、感知目标、规划轨迹、控制指令。

系统信息

诊断、日志和其他 ROS 话题消息。

rosbag 是 ROS 的录制与回放工具体系，MCAP 是 rosbag 可使用的一种存储格式，两者不是完全对立的概念。

ros2 bag info <MCAP数据目录或文件>
ros2 bag play <数据包路径>

06｜数据处理闭环与 Issue

目标是打通“数据上传—异常筛选—场景包制作—算法消费”的标准化链路。

完整数据链路

1｜车辆运行与录制
将关键 ROS 话题录制为 MCAP/rosbag 数据。
2｜Lichtblick 回放
查看图像、点云、Plot、Raw Messages，定位异常发生时间。
3｜异常筛选
优先保留影响自动驾驶的近处障碍物漏检；远处或对向车道且不影响驾驶的事件暂缓。
4｜切割场景包
按团队的数据时长规范切割，只保留复现异常所需的最小片段。
5｜上传并绑定 Issue
证据、数据和责任人统一进入工单入口，避免平台割裂。
6｜算法复现与反馈
算法人员消费数据、分析根因，并把结果回填到 Issue。

Issue 在这里代表什么

Issue 不只是“软件报错”，而是一条可持续跟踪的任务记录，它把一次异常从发现到关闭的必要材料绑定在一起：

字段内容
异常描述车辆表现、可能所属模块、影响程度。
时间与数据异常时间点、原始包和按规范生成的最小切割包。
证据截图、录屏、关键话题、波形或标注。
责任与状态认领人、处理进度、优先级和当前状态。
分析结论根因、修复方案、验证结果与是否关闭。

平台集成思路

以工单系统为统一入口：用户认领任务后，通过链接打开数据播放器，在回放同时完成证据标注和上传，所有数据最终回到同一个 Issue 中。

安全原则：操作车辆、设备、数据和账号前确认授权范围；遇到异常先停止可能扩大风险的操作，保留现场和日志，再上报处理。涉及密钥、密码和内部地址时不得公开上传。
